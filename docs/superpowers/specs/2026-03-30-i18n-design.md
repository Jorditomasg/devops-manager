# i18n — Internacionalización completa de DevOps Manager

**Fecha:** 2026-03-30
**Estado:** Aprobado
**Idiomas iniciales:** Español (`es_ES`), Inglés (`en`)

---

## Objetivo

Extraer **todos** los textos hardcodeados de la aplicación (labels, botones, tooltips, logs, mensajes de error, confirmaciones, placeholders, status text) a ficheros de traducción JSON, y añadir un selector de idioma en el diálogo de Configuración que aplica el cambio dinámicamente sin cerrar la app.

---

## Arquitectura

### Nuevos archivos

```
i18n/
  es_ES.json          ← idioma por defecto (español)
  en.json             ← inglés
core/
  i18n_manager.py     ← singleton: t(), set_language(), get_available_languages()
```

### `core/i18n_manager.py`

Módulo singleton importado como `from core.i18n_manager import t, set_language, get_available_languages`.

**API pública:**

- `t(key: str, **kwargs) -> str`
  Retorna el texto traducido para `key`. Si la clave no existe en el idioma activo, cae back al `es_ES`. Si tampoco existe, retorna `key` (nunca lanza excepción).
  Soporta interpolación via `str.format_map(kwargs)`: `t("log.pulling", count=3)` → `"[global] Descargando 3 repos..."`.

- `set_language(code: str) -> None`
  Carga `i18n/{code}.json`, actualiza el estado interno y persiste `code` en `devops_manager_config.json` bajo la clave `language`.

- `get_available_languages() -> dict[str, str]`
  Escanea `i18n/*.json`, lee la clave `_lang_name` de cada fichero, retorna `{"es_ES": "Español", "en": "English"}`. El nombre en el combo aparece en el propio idioma.

- `get_current_language() -> str`
  Retorna el código activo (ej. `"es_ES"`).

El módulo carga el idioma guardado en config al importarse por primera vez; si no hay config, usa `es_ES`.

## Ficheros JSON de traducción

### Estructura

Plana (sin anidamiento), con prefijos por contexto. La clave `_lang_name` es el nombre del idioma en su propio idioma.

**Prefijos:**
- `app.*` — título de ventana, nombre de la app
- `status.*` — textos de estado (stopped, running, starting, error, installing)
- `btn.*` — etiquetas de botones
- `tooltip.*` — tooltips de cualquier widget
- `label.*` — labels informativos y secciones
- `placeholder.*` — placeholder_text de entradas
- `log.*` — mensajes que van al panel de log
- `confirm.*` — preguntas de confirmación (messagebox.askyesno)
- `error.*` — mensajes de error (messagebox.showerror)
- `warn.*` — avisos (messagebox.showwarning)
- `info.*` — informativos (messagebox.showinfo)
- `settings.*` — textos del diálogo de configuración
- `dialog.*` — textos de otros diálogos (clone, profile, docker, config_editor…)
- `global_panel.*` — textos del panel global
- `card.*` — textos de las repo cards
- `tray.*` — textos del system tray

## Cambios en repo types YAML

En los 6 ficheros de `config/repo_types/*.yml`, la sección `ui.install` se normaliza así:

```yaml
install:
  label_missing: "Install"          # igual en todos
  label_ok: "Reinstall ✓"           # igual en todos
  status_label_deps_missing: "⚠ Falta instalar"  # igual en todos
  tooltip: "<comando real>"          # el install_cmd o reinstall_cmd del repo
```

El tooltip pasa a ser el comando exacto que se ejecutará (`npm i`, `mvn clean install -DskipTests -B`, etc.), no una descripción. La UI leerá este valor directamente del YAML como cadena técnica (no se traduce).

Los textos `label_missing`, `label_ok` y `status_label_deps_missing` los gestionará `t()` en el código Python directamente (`t("btn.install")`, `t("btn.reinstall")`, `t("label.status_deps_missing")`). El código **no leerá** estos valores del YAML para mostrarlos en la UI — los valores en el YAML quedan como documentación del valor canónico pero no se consumen como texto de interfaz.

## Selector de idioma en SettingsDialog

Nueva sección `_build_language_section(frame)` en `SettingsDialog`, añadida después de la sección de acceso rápido:

- Título: `"🌐 Idioma"` con descripción explicativa
- `CTkComboBox` con los nombres de idioma detectados (`get_available_languages()`)
- El combo muestra el nombre en su propio idioma: "Español", "English"
- Al guardar (`_save()`): se llama `set_language(selected_code)`, luego el callback `on_save` recibe los settings y opcionalmente incluye `language_changed=True`
- `app.py` detecta el cambio y llama a `_rebuild_ui()`: destruye y recrea cards + panel global + system tray menu

## Cambios en `gui/constants.py`

Se eliminan las constantes de texto:
- `NO_PROFILE_TEXT` → `t("label.no_profile")`
- `BTN_CONFIG_TEXT` → `t("btn.config")`
- `BTN_CONFIG_TOOLTIP` → `t("tooltip.config")`
- `REINSTALL_LBL` → `t("btn.reinstall")`

Se conservan todas las constantes no-texto (timings, regex, límites numéricos).

## Notas de implementación

- `t()` nunca lanza excepción — usa el key como fallback
- Los f-strings que mezclan texto traducible con valores dinámicos se convierten en `t("key", param=value)`
- El `i18n_manager` se inicializa antes de crear cualquier widget (al inicio de `main.py` o en el import)
- Los `ToolTip` se crean con `t()` en el momento de construcción del widget; al reconstruir la UI se recrean con el nuevo idioma
- Los `messagebox` siempre llaman `t()` en el momento de mostrarse, por lo que si el idioma cambia entre construcción y muestra del diálogo, mostrarán el idioma correcto
- El `_rebuild_ui()` de `app.py` primero cierra cualquier diálogo `CTkToplevel` abierto (iterando `winfo_children()`), luego destruye las cards y el panel global, y por último los recrea
